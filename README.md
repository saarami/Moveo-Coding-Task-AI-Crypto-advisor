# AI Crypto Advisor

A full-stack web application that gives each user a personalized daily crypto dashboard — powered by live market data, AI-generated analysis, and a feedback system that stores every vote for future recommendation improvements.

---

## User Flow

```
Sign up or log in
       ↓
Complete onboarding quiz (assets · investor type · content priorities)
       ↓
View personalized dashboard (prices · news · AI insight · meme)
       ↓
Vote thumbs up / down on each section
       ↓
Update preferences any time via the Preferences page
```

---

## Features

- **Authentication** — JWT-protected signup and login with bcrypt-hashed passwords
- **Onboarding** — three-question quiz: interested assets, investor type, content priorities
- **Editable Preferences** — dedicated `/preferences` page to update selections after onboarding
- **Personalized Dashboard** — four content sections assembled for the current user
- **Dashboard card ordering** — cards appear in the order set by the user's content priority preferences
- **Coin Prices** — live prices and 24 h change from CoinGecko, filtered to the user's tracked assets
- **Market News** — latest headlines from NewsData.io (static fallback when key is absent)
- **AI Insight** — 2-sentence market observation generated by OpenRouter → Hugging Face → static fallback
- **Crypto Meme** — random meme from a curated static list
- **Feedback Voting** — thumbs up / down on every section, stored per user per day in PostgreSQL
- **Saved Votes** — votes are restored on page refresh (fetched from the database on dashboard load)
- **Daily Dashboard Caching** — one snapshot per user per day stored in `daily_content`; external APIs are only called on cache miss
- **Preference-Driven Cache Invalidation** — saving updated preferences deletes today's cache so the next dashboard load uses the new asset selection
- **Docker Support** — PostgreSQL, backend, and frontend all runnable with a single `docker-compose up`

## AI Provider Fallback Chain

The AI Insight section uses a three-tier provider chain. On each dashboard cache miss, the backend attempts providers in order:

```
1. OpenRouter  (OPENROUTER_API_KEY)
   — tries multiple free models in sequence
   — moves to next provider if all models return errors or rate-limit

2. Hugging Face  (HUGGINGFACE_API_KEY)
   — OpenAI-compatible Inference Providers endpoint
   — model is configurable via HUGGINGFACE_MODEL

3. Static fallback
   — random insight from a curated list in fallback_data.py
   — always available, never fails
```

Every insight produced by a live provider is normalised to end with exactly one instance of "This is not financial advice." The app does not provide financial advice.

## Local Development Setup

### Option A — Full Stack with Docker Compose

Requires Docker Desktop. Starts PostgreSQL, backend, and frontend together.

```bash
docker-compose up -d --build
```

Services:

| Service | URL |
|---|---|
| Frontend | http://localhost:5173 |
| Backend API | http://localhost:8000 |
| Swagger UI | http://localhost:8000/docs |
| PostgreSQL | localhost:5433 |

To pass optional API keys to the Docker backend, create a root-level `.env` file (not committed):

```
OPENROUTER_API_KEY=sk-or-v1-...
NEWSDATA_API_KEY=...
SECRET_KEY=your-secure-64-char-secret
```

Docker Compose picks it up automatically.

Useful commands:

```bash
# View logs
docker-compose logs -f backend

# Stop everything
docker-compose down

# Rebuild after npm package changes
docker-compose build frontend
```

### Option B — Backend Locally, PostgreSQL via Docker

```bash
# 1. Start PostgreSQL only
docker-compose up -d postgres

# 2. Set up backend
cd backend
python -m venv .venv

# Windows
.venv\Scripts\activate
# macOS / Linux
source .venv/bin/activate

pip install -r requirements.txt
cp .env.example .env          # then edit DATABASE_URL and SECRET_KEY

# 3. Apply migrations
python -m alembic upgrade head

# 4. Start the backend
uvicorn app.main:app --reload
# → http://localhost:8000
```

### Option C — Frontend Locally

```bash
cd frontend
npm install
cp .env.example .env          # VITE_API_BASE_URL=http://localhost:8000
npm run dev
# → http://localhost:5173
```

## API Endpoints

Base path: `/api`

### Auth

| Method | Path | Description |
|---|---|---|
| `POST` | `/api/auth/register` | Create account, returns JWT |
| `POST` | `/api/auth/login` | Login, returns JWT |
| `GET` | `/api/auth/me` | Returns current user profile |

### Onboarding / Preferences

| Method | Path | Description |
|---|---|---|
| `GET` | `/api/onboarding/preferences` | Fetch saved preferences (404 if not set) |
| `POST` | `/api/onboarding/preferences` | Save or update preferences (upsert) |

### Dashboard

| Method | Path | Description |
|---|---|---|
| `GET` | `/api/dashboard` | Returns today's personalized dashboard snapshot |

The response includes `daily_content_id`, which is required for submitting feedback votes.

### Feedback

| Method | Path | Description |
|---|---|---|
| `POST` | `/api/feedback` | Submit or update a vote for a dashboard section |
| `GET` | `/api/feedback?daily_content_id=<id>` | Fetch all votes for a given daily snapshot |

All endpoints except `/api/auth/register`, `/api/auth/login`, and `/health` require:

```
Authorization: Bearer <token>
```

Interactive documentation is available at `http://localhost:8000/docs`.

## Database Tables

### `users`
Stores registered users. Fields: `id`, `name`, `email` (unique), `hashed_password`, `created_at`, `updated_at`.

### `user_preferences`
Stores onboarding answers. One row per user. Fields: `id`, `user_id`, `interested_assets`, `investor_type`, `content_types`, `created_at`, `updated_at`.

### `daily_content`
One dashboard snapshot per user per calendar day — the caching layer. Fields: `id`, `user_id`, `date`, `market_news`, `coin_prices`, `ai_insight`, `meme`, `created_at`. Unique constraint on `(user_id, date)`.

### `feedback`
One row per vote, per section, per day. Fields: `id`, `user_id`, `daily_content_id`, `section_type`, `content_item_id`, `vote`, `created_at`. Unique constraint on `(user_id, daily_content_id, section_type, content_item_id)` — re-voting updates the existing row instead of creating a duplicate.

---

## Feedback and Future Recommendations

Every thumbs up / down vote is stored in the `feedback` table and linked to the exact `daily_content` snapshot the user saw. This means a future recommendation model can read:

- Which assets the user was tracking
- Which investor type they identified as
- Which content sections they liked or disliked
- The exact content that was shown (via `daily_content`)

Possible future improvements:
- Weight AI insight prompts toward assets and topics the user consistently rates positively
- Reorder dashboard sections based on per-user vote history
- Use aggregated feedback across all users to surface trending content
- Retrain or fine-tune a ranking model on the accumulated signal

## Known Limitations and Future Improvements

| Area | Current State | Potential Improvement |
|---|---|---|
| NewsData.io news | Returns static fallback if `NEWSDATA_API_KEY` is not set | Add free-tier fallback via RSS |
| AI providers | Can be rate-limited; fallback returns static text | Add caching of AI responses per asset set |
| Rate limiting | No per-endpoint rate limiting in production | Add via middleware or API gateway |
| Email verification | Not implemented | Add email confirmation on signup |
| Financial advice | Not provided — disclaimer on every AI insight | N/A (by design) |
| Meme source | Static list in `fallback_data.py` | Integrate Reddit or Tenor API |
| Cache invalidation | Votes from earlier in the day are lost when preferences change | Preserve feedback across cache invalidation |
| Coin names | CoinGecko free `/simple/price` returns symbol only | Switch to `/coins/markets` for full names |
